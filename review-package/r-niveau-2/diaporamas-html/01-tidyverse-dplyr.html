Pourquoi interroger la base directement ?

Éviter les extractions lourdes et fragiles

  • Les données volumineuses restent côté serveur
  • Le filtrage se fait avant transfert
  • Moins de mémoire locale consommée
  • Meilleure traçabilité des requêtes
2 / 10

Connexion DBI

Pattern standard, quel que soit le SGBD

library(DBI)
library(RSQLite)

con <- dbConnect(RSQLite::SQLite(), "business.db")
dbListTables(con)

dbDisconnect(con)
3 / 10

Objets distants avec dbplyr

Code dplyr, exécution SQL

library(dplyr)
library(dbplyr)

ventes_tbl <- tbl(con, "ventes")

kpi <- ventes_tbl |>
  filter(date >= "2025-01-01") |>
  group_by(region) |>
  summarise(ca_total = sum(prix_net, na.rm = TRUE), .groups = "drop")

show_query(kpi)
kpi |> collect()
4 / 10

Lazy evaluation

La requête part au serveur au bon moment

  • tbl() crée une référence, pas une copie des données
  • Les verbes dplyr construisent la requête SQL
  • collect() déclenche l'exécution et rapatrie le résultat
  • Bonne pratique : collecter uniquement la table déjà agrégée
5 / 10

Bonnes pratiques de performance

  • Filtrer le plus tôt possible dans le pipeline
  • Limiter les colonnes avec select()
  • Éviter les jointures inutiles
  • Contrôler le SQL généré avec show_query()
  • Fermer proprement la connexion après traitement
6 / 10

Jointure distante (aperçu)

clients_tbl <- tbl(con, "clients")

analyse_tbl <- ventes_tbl |>
  left_join(clients_tbl, by = "client_id") |>
  group_by(region, segment) |>
  summarise(ca = sum(prix_net, na.rm = TRUE), .groups = "drop")
7 / 10

Points de vigilance

  • Types incohérents entre tables (texte vs numérique)
  • Colonnes dates mal parsées
  • Clés non uniques causant des duplications silencieuses
  • Droits SQL insuffisants pour certaines tables
