Chapitre 1 — Architecture modulaire

Concevoir un système Python évolutif et maintenable à long terme.

Objectifs architecture

  • Réduire le coût de changement fonctionnel.
  • Isoler les dépendances techniques.
  • Faciliter tests, observabilité et déploiements.
  • Clarifier les responsabilités par couche.

Découpage en couches

  • Domaine : règles métier et invariants.
  • Application : orchestration des cas d'usage.
  • Infrastructure : SQL, API, cache, bus.
  • Interface : HTTP, CLI, worker.

Ports et adaptateurs

from typing import Protocol

class OrderRepo(Protocol):
    def save(self, order) -> None: ...

class CreateOrder:
    def __init__(self, repo: OrderRepo) -> None:
        self.repo = repo

    def execute(self, order) -> None:
        self.repo.save(order)

Inversion de dépendances

  • Le domaine dépend d'abstractions, pas de SQLAlchemy ou FastAPI.
  • Le wiring des implémentations se fait au « composition root ».
  • Les tests remplacent facilement l'infrastructure par des doubles.

Organisation de code recommandée

src/
  domain/
  application/
  infrastructure/
  interfaces/
  bootstrap/
tests/

Frontières transactionnelles

class UnitOfWork(Protocol):
    def commit(self) -> None: ...
    def rollback(self) -> None: ...

class TransferService:
    def __init__(self, uow: UnitOfWork):
        self.uow = uow

    def execute(self):
        ...
        self.uow.commit()

Domain events

from dataclasses import dataclass

@dataclass(frozen=True)
class OrderPaid:
    order_id: str
    amount: float

Publier des événements découple les réactions secondaires (email, analytics...).

Gestion des erreurs par niveau

  • Domaine: exceptions métier explicites.
  • Application: mapping vers codes fonctionnels.
  • Infrastructure: wrapping des erreurs techniques.
  • Interface: réponse client claire et stable.

Configuration et secrets

  • Configuration centralisée et typée.
  • Aucun secret dans le code source.
  • Différencier config runtime et config build.
  • Tracer la version de configuration active.

Résilience: circuit breaker (notion)

class CircuitBreaker:
    def __init__(self, threshold: int = 3) -> None:
        self.threshold = threshold
        self.failures = 0

    def record(self, ok: bool) -> None:
        self.failures = 0 if ok else self.failures + 1

    @property
    def is_open(self) -> bool:
        return self.failures >= self.threshold

Monolithe modulaire vs microservices

  • Monolithe modulaire: meilleur point de départ pour de nombreuses équipes.
  • Microservices: utile si frontières métier et besoins d'échelle sont prouvés.
  • Le coût opérationnel doit guider le choix, pas la mode.

ADR: documenter les décisions

  • Contexte
  • Décision retenue
  • Alternatives rejetées
  • Conséquences et risques

Checklist de revue architecture

  • Les dépendances pointent-elles vers le domaine ?
  • Le code métier est-il testable sans I/O externe ?
  • Les responsabilités sont-elles lisibles en 5 minutes ?
  • Les risques techniques sont-ils explicités ?

Récapitulatif

  • Une bonne architecture ralentit la dette et accélère l'évolution.
  • Ports/adaptateurs et frontières claires sont les bases Python robustes.
  • La qualité d'architecture se valide en production, pas en diagramme seul.
