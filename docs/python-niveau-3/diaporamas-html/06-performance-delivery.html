Chapitre 6 — Performance et delivery

Optimiser sans régression et livrer de façon contrôlée.

Principe directeur

  • Mesurer avant d'optimiser.
  • Prioriser les points chauds métier.
  • Documenter chaque arbitrage technique.
  • Relier performance à la qualité de service utilisateur.

Budgets de performance

  • Latence p95 API: cible explicite.
  • Débit minimal supporté.
  • Consommation mémoire maximale.
  • Temps batch maximal.

Profilage CPU

import cProfile
import pstats

with cProfile.Profile() as prof:
    run_pipeline()

pstats.Stats(prof).sort_stats("cumtime").print_stats(20)

Profilage mémoire

import tracemalloc

tracemalloc.start()
run_pipeline()
current, peak = tracemalloc.get_traced_memory()
print(f"current={current/1e6:.1f}MB peak={peak/1e6:.1f}MB")
tracemalloc.stop()

Complexité algorithmique

  • Éviter les boucles imbriquées O(n²) non nécessaires.
  • Préférer dictionnaires/sets pour recherches fréquentes.
  • Réduire les conversions répétées de structures.

Choix de parallélisme

  • I/O dominant: asyncio.
  • CPU dominant: multiprocessing / worker pool.
  • Code bloquant isolé: asyncio.to_thread.
  • Toujours mesurer le coût de coordination.

Cache applicatif

from functools import lru_cache

@lru_cache(maxsize=512)
def fx_rate(base: str, quote: str) -> float:
    # appel externe simulé
    return 1.08

Le cache doit avoir une politique d'invalidation claire.

Batching et réduction des I/O

  • Regrouper lectures/écritures pour réduire overhead réseau.
  • Limiter les allers-retours DB par requête.
  • Utiliser upsert/bulk insert quand possible.

Optimisation data Python

# anti-pattern: boucle Python
# for row in rows: total += row["amount"]

# meilleur: agrégation vectorisée (pandas/NumPy)
# total = df["amount"].sum()

Load tests et scénarios

  • Tester charge nominale + pic + dégradation.
  • Mesurer p50/p95/p99, erreurs et saturation.
  • Conserver l'historique des campagnes de charge.

Pipeline CI/CD pragmatique

1) Lint + type + unit tests
2) Build artefact versionné
3) Deploy staging
4) Smoke tests
5) Déploiement progressif production

Release progressive et feature flags

  • Canary: exposer progressivement à un sous-ensemble d'utilisateurs.
  • Feature flags: activer/désactiver sans redéploiement complet.
  • Surveiller métriques business et techniques après release.

Plan de rollback

  • Version précédente disponible et testée.
  • Critères de déclenchement rollback connus.
  • Responsable décisionnel identifié.
  • Runbook incident prêt et partagé.

Récapitulatif

  • Performance durable = mesure, ciblage, validation.
  • Delivery fiable = pipeline, canary, rollback maîtrisé.
  • L'objectif final reste la stabilité de service métier.
