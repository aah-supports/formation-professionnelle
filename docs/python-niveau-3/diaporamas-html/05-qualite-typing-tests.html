Chapitre 5 — Qualité, typing et tests

Sécuriser le comportement logiciel avant la production.

Dimensions de qualité

  • Correctness: résultat conforme au besoin métier.
  • Robustesse: comportement stable en cas d'erreur.
  • Maintenabilité: code compréhensible et modifiable.
  • Observabilité: incidents détectables rapidement.

Typing explicite

from typing import TypedDict

class OrderRow(TypedDict):
    order_id: str
    amount: float
    country: str


def normalize(row: OrderRow) -> OrderRow:
    row["country"] = row["country"].upper()
    return row

Protocol pour contrats internes

from typing import Protocol

class Notifier(Protocol):
    def send(self, target: str, message: str) -> None: ...


def notify_invoice(notifier: Notifier, email: str) -> None:
    notifier.send(email, "Facture prête")

Unit tests ciblés (AAA)

def test_total_panier():
    # Arrange
    lignes = [10.0, 15.0, 5.0]
    # Act
    total = sum(lignes)
    # Assert
    assert total == 30.0

Tests d'intégration utiles

  • Vérifier contrats entre modules (repo, API, queue).
  • Utiliser jeux de données réalistes mais limités.
  • Mesurer le temps d'exécution pour garder un feedback rapide.

  • Complète les tests scénarisés en explorant plus de cas.

Contract tests pour adaptateurs

  • Définir le comportement attendu d'un port.
  • Valider chaque implémentation contre ce contrat.
  • Réduit les surprises lors du changement d'infrastructure.

Mutation testing (principe)

  • Modifier automatiquement le code pour simuler des bugs.
  • Mesurer si les tests détectent ces mutations.
  • Indique la force réelle de la suite de tests.

Qualité en CI

  • Pipeline court: lint + typing + tests rapides sur chaque PR.
  • Pipeline étendu: intégration/perf sur branche principale.
  • Bloquer le merge en cas d'échec d'un garde-fou critique.

Signaux de dette qualité

  • Tests instables (flaky) ignorés régulièrement.
  • Correction manuelle récurrente en production.
  • Taux de rollback élevé après release.
  • Temps d'analyse incident trop long.

Checklist avant merge

  • Le diff est-il couvert par au moins un test pertinent ?
  • Les erreurs et cas limites sont-ils traités ?
  • Le typage améliore-t-il la lisibilité API ?
  • Les logs facilitent-ils le diagnostic futur ?

Récapitulatif

  • Qualité = typing + tests + instrumentation, pas un seul outil.
  • La suite de tests doit protéger les flux critiques métier.
  • La CI transforme ces exigences en discipline collective.
